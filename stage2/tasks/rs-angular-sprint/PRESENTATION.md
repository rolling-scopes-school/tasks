# Презентация проекта

Финальная презентация состоит из двух частей, которые проходят в рамках одного события.

## Участники

- **Команда** — участие всех обязательно
- **Жюри** — менторы и активисты, знакомятся с README и артефактами **заранее**
- Все сессии **открыты** — любой студент или ментор может присоединиться как наблюдатель

---

## Часть 1: Командная (~30 мин)

**Цель:** показать продукт и рассказать историю команды — как работали вместе, как изучали Angular, какие решения принимали.

| Часть            | ~Время    | Что происходит                                                                                                          |
| ---------------- | --------- | ----------------------------------------------------------------------------------------------------------------------- |
| **Product Demo** | 10-15 мин | Команда показывает приложение (live или видео) и подсвечивает технические решения: DI, CD, Forms, Routing, Signals.     |
| **Team Story**   | 10 мин    | Как работали: роли, процесс изучения Angular, конфликты, решения. Что далось тяжелее всего?                             |
| **Q&A**          | 5-10 мин  | Вопросы жюри по архитектуре, коду и Angular-специфичным решениям.                                                       |

### Рекомендации

- По возможности включите веб камеры
- Слайды рекомендуются, но не обязательны. Вся информация может быть в README
- Приоритет у live-демо. Показывать видео — только если live-показ сорвался
- Покажите Angular DevTools: дерево компонентов, Change Detection profiler

**Оценка:** [SCORE_TEAM.md](./SCORE_TEAM.md) (260 баллов, одинаково для всей команды).

---

## Часть 2: Соло (~20 мин на студента)

**Цель:** показать личный вклад в проект и глубину понимания Angular.

Каждый студент выступает **индивидуально** после командной части. Жюри — те же.

| Этап               | ~Время  | Что происходит                                                                                               |
| ------------------ | ------- | ------------------------------------------------------------------------------------------------------------ |
| **Личный вклад**   | 5-7 мин | Студент рассказывает: что именно он реализовал, какие Angular-концепции применял, с какими проблемами столкнулся. |
| **Q&A**            | 10 мин  | Вопросы по коду студента и базовым Angular-концепциям. Студент может показывать код.                         |
| **Оценка**         | 3 мин   | Жюри заполняет форму.                                                                                        |

### Что спросят

- Конкретные решения из кода студента: «Почему здесь `OnPush`?», «Как устроен этот сервис?», «Что изменится, если убрать этот guard?»

### Подготовка студента

1. Подготовить вступление — ваша роль, **2-3 самые сложные Angular-проблемы**, с которыми столкнулись
2. Знать весь код, над которым пришлось работать — уметь объяснить любое решение, архитектурный выбор или реализацию
3. Дневник замержен в `main` до начала презентации

### Оценка (50 баллов)

Жюри выбирает **один из четырёх уровней** и пишет обоснование (1-2 предложения).

| Уровень       | Баллы | Индикаторы                                                                                                                    |
| ------------- | ----- | ----------------------------------------------------------------------------------------------------------------------------- |
| **Weak**      | 10    | Не может объяснить свой код или базовые Angular-концепции (DI, CD). Нет структуры рассказа.                                   |
| **OK**        | 25    | Понимает DI и CD на базовом уровне. Отвечает на прямые вопросы, но без контекста. Рассказ — «по вопросам».                    |
| **Good**      | 40    | Уверенный рассказ со структурой. Объясняет архитектурные решения и trade-offs. Спокойно реагирует на вопросы.                 |
| **Excellent** | 50    | Рассказ как история: проблема → решение → результат. Понимает внутреннее устройство Angular (Zone.js, инжекторы, дерево CD).  |

> **Red Flag:** если студент не может объяснить свой код или базовые концепции (DI, CD) — максимум 10 баллов.

### Подготовка жюри

Перед началом соло-части (5-10 мин):

1. Открыть **self-assessment PR** студента — посмотреть, что он заявляет о своём вкладе
2. Ознакомиться с критериями оценки [выше](#оценка-50-баллов)

---

> По решению жюри время на команду или на студента может быть увеличено.
