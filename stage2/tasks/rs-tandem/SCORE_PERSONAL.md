# Personal Score (500 + 100 баллов)

Личная оценка каждого студента. Этот документ определяет два Score в RS APP: **Tandem: Personal** (500 pts) и **Tandem: Peer Review** (100 pts).

**Поток оценки:**

1. Студент заявляет фичи с баллами в [self-assessment](#self-assessment)
2. [Студенты-эксперты](./DEFENCE_PEER.md) валидируют: смотрят демо и отмечают **расхождения** с заявленным (фича не работает, завышена, не продемонстрирована). Всё, что не оспорено — считается подтверждённым.
3. [Ментор](./DEFENCE_MENTOR.md) получает self-assessment + результат peer review. Оценивает качество кода и понимание — не перепроверяет "работает ли".
4. **Координатор** выставляет итоговый балл Personal Features на основе self-assessment, скорректированного по замечаниям peers. Ментор не влияет на Personal Features — только на Mentor Review (150 pts).

### Порог допуска

Без выполнения этих требований студент **не допускается** к оценке:

- Дневник разработки существует, заполнен регулярно, записи закоммичены своевременно
- Описан в дневнике или снят на видео процесс разработки минимум 2 личных Feature Component

## Self-Assessment

Перед защитами студент создаёт **PR** в репозитории команды с файлом `self-assessment.md` в папке `/development-notes/github-username/`.

Каждая заявленная фича — со ссылкой на файл или PR:

```markdown
### Complex Component: Game Board (+25) ★

- Файл: src/components/GameBoard/
- PR: #42
- Что делает: интерактивная доска с drag&drop, WebSocket синхронизация
```

Отметьте **★** у 2-3 ключевых фич — их вы будете демонстрировать на [Peer Review](./DEFENCE_PEER.md) и [менторской сессии](./DEFENCE_MENTOR.md).

Self-assessment используется дважды:

- **Peer Review** — студенты-эксперты верифицируют: фичи работают? Заявленное реально?
- **Менторская сессия** — ментор смотрит код фич со ★ и оценивает архитектуру.

## 1. Development Diary (100 баллов)

**Оценивается автоматически** AI-агентом (Tandi) до начала защит. Ментор не оценивает дневник — он использует его как контекст на [менторской сессии](./DEFENCE_MENTOR.md).

Подробные правила: [DEVELOPMENT_DIARY.md](./DEVELOPMENT_DIARY.md)

| Баллы   | Уровень      | Описание                                                                              |
| ------- | ------------ | ------------------------------------------------------------------------------------- |
| **20**  | Формальный   | Краткие записи без деталей ("сделал X")                                               |
| **60**  | Регулярный   | Минимум 2 записи/неделю, виден прогресс, описаны 2 компонента                         |
| **80**  | Детальный    | Есть описание проблем и решений, понятен ход мысли                                    |
| **100** | Рефлексивный | Глубокая рефлексия, сравнение вариантов реализации, видно что написано не для галочки |

## 3. Peer Review (100 баллов)

**Score в RS APP:** Tandem: Peer Review (отдельный Score).

Студенты-эксперты оценивают по 4 критериям на [сессии Peer Review](./DEFENCE_PEER.md) (~30 мин).

Всё заявленное в self-assessment **считается подтверждённым по умолчанию**. Эксперты отмечают только расхождения — фичи, которые не работают, выглядят завышенными или не были продемонстрированы. Оценку эксперты заносят в **Google Form** (приватная, не в PR).

| Критерий                 | Баллы  | Что проверяют                                                                       |
| ------------------------ | ------ | ----------------------------------------------------------------------------------- |
| **Deploy & Launch**      | **15** | Приложение задеплоено или запускается локально. Ссылка из README работает.          |
| **Feature Verification** | **50** | Студент демонстрирует фичи из self-assessment. Заявленные фичи реально работают.    |
| **Product Quality**      | **20** | User flow понятен. Edge cases обработаны (404, loading, ошибки).                    |
| **README & Artifacts**   | **15** | README содержит деплой-ссылку, Feature List, демо-видео. Self-assessment PR создан. |

- Каждый из 3 экспертов ставит оценку независимо (0-100).
- Финальная оценка = **медиана** трёх оценок.
- Если разброс > 40 баллов — координатор проверяет.

## 4. Mentor Review (150 баллов)

Субъективная оценка ментора по **трём критериям** (по 50 баллов) на [менторской сессии](./DEFENCE_MENTOR.md). Для каждого критерия ментор выбирает **один из четырёх уровней** (10 / 25 / 40 / 50) и пишет **обоснование (1-2 предложения)**.

### Technical Depth (50 pts)

| Уровень       | Баллы | Индикаторы (что наблюдает ментор)                                                                                 |
| ------------- | ----- | ----------------------------------------------------------------------------------------------------------------- |
| **Weak**      | 10    | Простые задачи, стандартные решения из туториалов. Не может объяснить, почему выбрано именно это решение.         |
| **OK**        | 25    | Решена хотя бы одна нетривиальная задача. Понимает "почему так". Может назвать альтернативы.                      |
| **Good**      | 40    | Работает со сложной технологией (WebSocket, AI, Canvas, custom backend). Объясняет trade-offs выбранного подхода. |
| **Excellent** | 50    | Глубокое владение технологией. Объясняет ограничения, edge cases, что бы сделал иначе при повторной реализации.   |

> **Обоснование (обязательно):** ментор пишет 1-2 предложения, объясняющие выбранный уровень.

### Code Architecture (50 pts)

| Уровень       | Баллы | Индикаторы (что наблюдает ментор)                                                                      |
| ------------- | ----- | ------------------------------------------------------------------------------------------------------ |
| **Weak**      | 10    | Хаотичная структура. Бизнес-логика в компонентах. Нет разделения ответственности.                      |
| **OK**        | 25    | Базовая структура есть. Выделен API-слой. Компоненты не гигантские.                                    |
| **Good**      | 40    | Чистое разделение: UI / логика / API. Переиспользуемые хуки/утилиты. Читаемый код.                     |
| **Excellent** | 50    | Продуманная архитектура. Паттерны применены осознанно. Код легко расширять. TypeScript strict без any. |

> **Обоснование (обязательно):** ментор пишет 1-2 предложения, объясняющие выбранный уровень.

### Demo & Communication (50 pts)

| Уровень       | Баллы | Индикаторы (что наблюдает ментор)                                                                                               |
| ------------- | ----- | ------------------------------------------------------------------------------------------------------------------------------- |
| **Weak**      | 10    | Не может объяснить свой код. Путается в ответах. Демо не подготовлено.                                                          |
| **OK**        | 25    | Показывает работу. Отвечает на прямые вопросы. Но не объясняет "почему".                                                        |
| **Good**      | 40    | Уверенная презентация. Объясняет решения и альтернативы. Отвечает на неожиданные вопросы.                                       |
| **Excellent** | 50    | Отличная структура рассказа. Объясняет trade-offs и ограничения. Видно глубокое понимание всего проекта, не только своей части. |

> **Обоснование (обязательно):** ментор пишет 1-2 предложения, объясняющие выбранный уровень.

### Red Flag

| Red Flag                    | Влияние                          |
| --------------------------- | -------------------------------- |
| Не может объяснить свой код | Demo & Communication: max 10 pts |
