# Personal Score (500 баллов)

> **Score в RS APP:** Personal Score  
> **Тип оценки:** личная (у каждого студента в команде своя оценка)  
> **Когда:** индивидуальный созвон после сдачи проекта  
> **Кто оценивает:** рандомно назначенный ментор

---

## Структура

```
Personal Score (500 pts max)
├── Development Diary        100 pts
├── Personal Features         200 pts (ceiling)    <-- ЛИЧНЫЙ чеклист
└── Mentor Review           200 pts               <-- расширенная оценка
```

---

## 1. Development Diary (100 баллов)

Подробные правила ведения дневника: [DEVELOPMENT_DIARY.md](./DEVELOPMENT_DIARY.md)

### Порог допуска (pass/fail)

Без выполнения этих требований студент **не допускается** к оценке Personal Score:

| Требование | Статус |
|------------|--------|
| Дневник разработки существует, заполнен регулярно, записи закоммичены своевременно | must pass |
| Описан в дневнике или снят на видео процесс разработки минимум 2 личных Feature Component | must pass |

### Критерии качества (0-100 баллов)

| Баллы | Уровень | Описание |
|-------|---------|----------|
| **20** | Формальный | Краткие записи без деталей ("сделал X") |
| **60** | Регулярный | Минимум 2 записи/неделю, виден прогресс, описаны 2 компонента |
| **80** | Детальный | Есть описание проблем и решений, понятен ход мысли |
| **100** | Рефлексивный | Глубокая рефлексия, сравнение вариантов реализации, видно что написано не для галочки |

## 2. Personal Features (200 баллов, ceiling)

**Это ЛИЧНЫЙ чеклист.** Студент заявляет только фичи, которые **он лично реализовал** (или значительно к ним причастен).

### Правила

1. Каждую фичу может заявить **только один студент** в команде (или два, если реально работали вместе)
2. На демо ментор верифицирует: "покажи код этого компонента"
3. Общая сумма может превысить 200, но в зачёт идёт максимум **200 баллов**
4. Это "меню выбора" — выбирайте то, что сделали именно вы

## 3. Mentor Review (200 баллов)

Субъективная оценка ментора по **четырём критериям**. Для каждого критерия ментор выбирает **один из четырёх уровней** и пишет **обоснование (1-2 предложения)**.

### Technical Depth (50 pts)

| Уровень | Баллы | Индикаторы (что наблюдает ментор) |
|---------|-------|-----------------------------------|
| **Weak** | 10 | Простые задачи, стандартные решения из туториалов. Не может объяснить, почему выбрано именно это решение. |
| **OK** | 25 | Решена хотя бы одна нетривиальная задача. Понимает "почему так". Может назвать альтернативы. |
| **Good** | 40 | Работает со сложной технологией (WebSocket, AI, Canvas, custom backend). Объясняет trade-offs выбранного подхода. |
| **Excellent** | 50 | Глубокое владение технологией. Объясняет ограничения, edge cases, что бы сделал иначе при повторной реализации. |

> **Обоснование (обязательно):** ментор пишет 1-2 предложения, объясняющие выбранный уровень.

### Code Quality (50 pts)

| Уровень | Баллы | Индикаторы (что наблюдает ментор) |
|---------|-------|-----------------------------------|
| **Weak** | 10 | Много `any`, `console.log`, необработанные ошибки. Код сложно читать. Нет структуры. |
| **OK** | 25 | Код читаемый. Есть структура папок/модулей. Базовая обработка ошибок (try/catch). |
| **Good** | 40 | Чистый код. Осмысленные абстракции. Обработка ошибок с user-friendly сообщениями. Код тестируемый. |
| **Excellent** | 50 | Паттерны проектирования к месту. Разделение ответственности. Документация ключевых решений. Нет "мёртвого" кода. |

> **Обоснование (обязательно):** ментор пишет 1-2 предложения, объясняющие выбранный уровень.

### Product & UX (50 pts)

| Уровень | Баллы | Индикаторы (что наблюдает ментор) |
|---------|-------|-----------------------------------|
| **Weak** | 10 | Базовый функционал. Нет loading states. Ошибки не отображаются пользователю. UX запутанный. |
| **OK** | 25 | Основной flow работает. Есть loading states. Интуитивно понятно, куда нажимать. |
| **Good** | 40 | Приятно пользоваться. Анимации/переходы. Empty states. Обработка edge cases (нет данных, ошибка сети). |
| **Excellent** | 50 | "Вау-эффект". Продуманные детали (микро-взаимодействия, accessibility). Хочется пользоваться снова. |

> **Обоснование (обязательно):** ментор пишет 1-2 предложения, объясняющие выбранный уровень.

### Demo & Communication (50 pts)

| Уровень | Баллы | Индикаторы (что наблюдает ментор) |
|---------|-------|-----------------------------------|
| **Weak** | 10 | Не может объяснить свой код. Путается в ответах. Демо не подготовлено. |
| **OK** | 25 | Показывает работу. Отвечает на прямые вопросы. Но не объясняет "почему". |
| **Good** | 40 | Уверенная презентация. Объясняет решения и альтернативы. Отвечает на неожиданные вопросы. |
| **Excellent** | 50 | Отличная структура рассказа. Объясняет trade-offs и ограничения. Видно глубокое понимание всего проекта, не только своей части. |

> **Обоснование (обязательно):** ментор пишет 1-2 предложения, объясняющие выбранный уровень.

### Red Flags (снижение Mentor Review)

| Red Flag | Влияние |
|----------|---------|
| Не может объяснить свой код | Demo & Communication: max 10 pts |
| Огромные файлы (500+ строк) | Code Quality: понижение на один уровень |
| Copy-paste код | Code Quality: понижение на один уровень |
| Заявленная фича не работает на демо | Вычитается из Personal Features |

## Процесс оценки

### Формат

Индивидуальный созвон студента с рандомно назначенным ментором.

**Длительность:** ~15-20 минут на студента.

### Кто участвует

- **Студент** — показывает свою работу
- **Ментор** — рандомно назначенный из пула менторов курса (аналогично corejs-интервью)

### Структура созвона (~15-20 мин)

| Этап | Время | Описание |
|------|-------|----------|
| **Дневник** | 3-5 мин | Студент кратко рассказывает о своем пути: что изучил, какие проблемы решал. Ментор просматривает дневник. |
| **Feature Demo** | 7-10 мин | Студент показывает свои личные Feature Components. Демонстрирует работу в приложении и показывает код. Ментор отмечает фичи в чеклисте. |
| **Q&A** | 3-5 мин | Ментор задает вопросы по коду и архитектуре. Верифицирует, что студент понимает свой код. |
| **Оценка** | 2 мин | Ментор заполняет Scoring Sheet. Для Mentor Review выбирает уровень (10/25/40/50) и пишет обоснование по каждому критерию. |

### Подготовка студента к созвону

1. **Дневник** должен быть смержен в один файл (`username-merged.md`)
2. **Заполнить Personal Features чеклист** — заранее отметить, что вы лично реализовали (самооценка)
3. **Подготовить демо** своих 2+ Feature Component (знать, что показать и рассказать)
4. **Быть готовым показать код** и объяснить, почему выбрали именно такое решение

### Подготовка ментора к созвону

1. Открыть этот документ (PERSONAL_SCORE.md)
2. Ознакомиться с **дискретными уровнями Mentor Review** (Weak/OK/Good/Excellent) и их индикаторами
3. Перед созвоном: бегло просмотреть репозиторий студента и дневник (~5 мин)
4. Посмотреть самооценку студента (заполненный Personal Features чеклист)
5. Во время созвона: заполнять Scoring Sheet по ходу демо
6. **Верифицировать заявленные фичи:** "покажи код", "объясни, как это работает"
7. **Mentor Review:** для каждого критерия выбрать уровень и написать обоснование (1-2 предложения)
8. После созвона: выставить оценку в RS APP
