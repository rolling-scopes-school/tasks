# Менторская сессия

Индивидуальный созвон студента с рандомно назначенным ментором.

**Цель:** показать **глубину** — насколько студент разобрался в использованных технологиях, инструментах и архитектурных решениях. [Personal Features](./SCORE_PERSONAL.md#personal-features-250-баллов) отвечают за широту (сколько сделал), менторская сессия — за глубину (насколько понял).

Это подготовка к одному из самых частых вопросов на интервью: _«Расскажите о самом сложном, что вы делали в последнее время»._ Здесь студент рассказывает про баги и как их фиксил, какие технологии выбрал и почему, какие trade-offs взвешивал.

Ментор **не проверяет "работает ли"** — это уже сделали студенты-эксперты на [Peer Review](./DEFENCE_PEER.md). Ментор фокусируется на коде и понимании.

## Формат (~30 мин)

| Этап                  | Время   | Описание                                                                                                    |
| --------------------- | ------- | ----------------------------------------------------------------------------------------------------------- |
| **О проекте**         | 3 мин   | Студент даёт контекст: что за приложение, его роль в команде.                                               |
| **Дневник**           | 5 мин   | Студент кратко рассказывает о своём пути: что изучил, какие проблемы решал. Ментор использует как контекст. |
| **Code Review + Q&A** | 15+ мин | Студент показывает код фич со ★ из self-assessment, объясняет архитектуру. Ментор задаёт вопросы по ходу.   |
| **Оценка**            | 2 мин   | Ментор заполняет Google Form с обоснованием.                                                                |

## Оценка (150 баллов)

Субъективная оценка ментора по **двум критериям**. Для каждого ментор выбирает **один из четырёх уровней** и пишет **обоснование (1-2 предложения)**.

### Technical Depth & Architecture (+100)

| Уровень       | Баллы | Индикаторы (что наблюдает ментор)                                                                                                                                                  |
| ------------- | ----- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Weak**      | 20    | Стандартные решения из туториалов. Код работает, но студент не может объяснить, почему выбрано именно это решение. Нет осознанной структуры.                                       |
| **OK**        | 50    | Решена хотя бы одна нетривиальная задача. Есть осознанное разделение ответственности. Понимает «почему так», может назвать альтернативы.                                           |
| **Good**      | 80    | Работает с технологией на уровне глубже «подключил и заработало». Чистая архитектура, понятная другому разработчику. Объясняет trade-offs.                                         |
| **Excellent** | 100   | Решал нестандартные проблемы, которых нет в документации. Продуманная архитектура с обоснованием. Объясняет ограничения, edge cases, что бы сделал иначе при повторной реализации. |

> **Примеры «Good»:** custom WebSocket-протокол с reconnect-логикой; RAG-пайплайн с chunking-стратегией; Canvas game loop с оптимизацией рендера; осмысленные тесты на edge cases; обоснованный выбор библиотеки (сравнил варианты); оптимизация узкого места после профилирования.

### Communication (+50)

| Уровень       | Баллы | Индикаторы (что наблюдает ментор)                                                                                               |
| ------------- | ----- | ------------------------------------------------------------------------------------------------------------------------------- |
| **Weak**      | 10    | Не может объяснить свой код. Путается в ответах.                                                                                |
| **OK**        | 25    | Показывает работу. Отвечает на прямые вопросы. Но не объясняет «почему».                                                        |
| **Good**      | 40    | Уверенная презентация. Объясняет решения и альтернативы. Отвечает на неожиданные вопросы.                                       |
| **Excellent** | 50    | Отличная структура рассказа. Объясняет trade-offs и ограничения. Видно глубокое понимание всего проекта, не только своей части. |

> **Red Flag:** если студент не может объяснить свой код — Communication: max 10 pts.

## Подготовка студента

1. **Self-assessment PR** должен быть создан ([подробности](./SCORE_PERSONAL.md#self-assessment))
2. **Дневник** замержен в `main` (файл `username-merged.md`)
3. **Подготовить "О проекте"** — 2-3 предложения: что за приложение, ваша роль
4. **Знать код** фич со ★ — где что лежит, уметь объяснить архитектуру
5. **Быть готовым к вопросам** по коду, техническим решениям и чекпоинтам (W1-W5)

## Подготовка ментора

1. Ознакомиться с критериями [выше](#оценка-150-баллов)
2. Перед созвоном (~5 мин): открыть self-assessment PR, посмотреть оценку и замечания Peer Review (если есть расхождения — обратить внимание на эти фичи)
3. Бегло просмотреть репозиторий и дневник
4. На созвоне: **фокус на коде** — продукт уже проверен peers, не нужно тестировать user flow
5. После созвона: заполнить Google Form
