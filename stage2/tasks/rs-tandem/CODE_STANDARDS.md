## Часть A — Код (50 баллов)

#### 1. TypeScript Strict (+15)

1. (+5) TypeScript используется, в `tsconfig.json` включён `"strict": true`.
2. (+5) В коде нет `@ts-ignore`.
3. (+5) В коде нет `as any` (приведение к any для обхода типизации).

**Как проверить:** открыть `tsconfig.json`, затем выполнить поиск по проекту:

```bash
grep -r "@ts-ignore\|as any" src/
```

#### 2. No Any (+15)

1. (+15) В коде нет явного использования типа `any` (`: any`, `<any>`).
2. (-5) 1-10 использований `any` — частичный балл (10 из 15).
3. (-15) Более 10 использований `any` — 0 баллов за эту секцию.

**Как проверить:**

```bash
grep -rn ": any\|<any>" src/ --include="*.ts" --include="*.tsx" | wc -l
```

> **Исключение:** `any` в типах внешних библиотек (если нет `@types`) — не штрафуется при наличии комментария `// TODO: replace any`.

#### 3. Linting Setup (+10)

1. (+5) ESLint настроен: файл конфигурации `.eslintrc.*` или `eslint.config.*` существует, `npm run lint` выполняется без ошибок.
2. (+5) Prettier настроен: файл `.prettierrc` или `prettier.config.*` существует, код отформатирован единообразно.

**Как проверить:**

```bash
npm install && npm run lint
```

Убедиться, что нет ошибок линтинга. Проверить наличие файлов конфигурации.

> **Про Husky.** Наличие `.husky/` в репозитории само по себе не показатель — часто настраивают для галочки. Реальная ценность git hooks видна по результату: консистентные коммит-сообщения, нет broken builds в main, код отформатирован единообразно. Эти признаки уже покрываются критериями Linting (Prettier) и History Hygiene (Teamwork). Отдельные баллы за Husky убираем — если команда реально его использует, это будет видно по качеству истории.

#### 4. Error Handling (+10)

1. (+5) API-запросы обёрнуты в try/catch (проверяется по коду).
2. (+5) Есть UI для отображения ошибок — toast, error page, alert (проверяется в продукте, Часть B п.3).

**Как проверить:** spot-check API-вызовы в коде — обёрнуты в try/catch?

> **Примечание:** п.1 проверяется здесь (код), п.2 верифицируется в Части B «Обработка edge cases». Баллы не дублируются: здесь — наличие try/catch в коде, там — результат для пользователя (видит ли он ошибку).

## Часть B — Продукт (50 баллов)

#### 1. Деплой или запуск (+10)

Деплой доступен **ИЛИ** проект запускается на созвоне с командой.

**Как проверить:** открыть URL из README. Если деплоя нет — связаться с командой, договориться о созвоне (~15 мин), команда запускает проект, вы проверяете.

#### 2. Основной user flow (+20)

Основной user flow проходим от начала до конца по Feature List из README.

**Как проверить:** прочитать Feature List в README, пройти каждый описанный сценарий. Всё работает — 20. Работает частично — 10. Не работает — 0.

#### 3. Обработка edge cases (+10)

404, loading state, обработка ошибок в UI.

**Как проверить:**

- `/nonexistent` — есть 404-страница?
- DevTools → Network → Slow 3G — есть loading states?
- DevTools → Network → Offline → действие, требующее сети — есть сообщение об ошибке?

#### 4. README (+10)

README содержит: деплой-ссылку или инструкцию запуска, Feature List, демо-видео.

**Как проверить:** открыть README, убедиться что всё на месте.

## Инструкции для кросс-чекера

### Часть A — Код (50 pts)

1. Клонировать репозиторий
2. Открыть `tsconfig.json` → `"strict": true`?
3. `grep -r "@ts-ignore\|as any" src/` → считаем
4. `grep -rn ": any\|<any>" src/ --include="*.ts" --include="*.tsx" | wc -l`
5. Проверить наличие ESLint + Prettier конфигов, запустить `npm run lint`
6. Spot-check: API-вызовы обёрнуты в try/catch?

### Часть B — Продукт (50 pts)

1. Открыть деплой из README. **Если деплоя нет** → связаться с командой, договориться о созвоне (~15 мин). Команда запускает проект, вы проверяете. **Созвон обязателен** — не пропускайте команду без проверки.
2. Прочитать Feature List в README
3. Пройти основной user flow по Feature List
4. Проверить 404 (`/random-url`), loading (DevTools → Slow 3G), ошибки (Network → Offline)
5. Если есть демо-видео — использовать как ориентир

> **Нет деплоя и команда не выходит на связь?** Напишите координаторам курса. Не ставьте 0 без попытки связаться.

## Что нужно от команды

- **Feature List в README** — 3-5 фич с шагами: _"Quiz mode: Open → Start → Answer 3 questions → See score"_

- **Демо-видео (до 8 мин)** — обязательно для всех
