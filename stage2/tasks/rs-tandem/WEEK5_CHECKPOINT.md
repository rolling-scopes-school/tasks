## Зачем это задание

Обработка ошибок — то, что отличает учебный проект от продукта. Пользователь не должен видеть белый экран, бесконечный спиннер или стектрейс в консоли. Три базовых сценария:

1. **Несуществующая страница** → понятная 404 страница с навигацией обратно.
2. **Загрузка данных** → loading state (спиннер, skeleton, текст) вместо пустоты.
3. **Ошибка сети / API** → понятное сообщение, а не краш приложения.

Чекпоинт проверяет, что ваше приложение корректно обрабатывает эти три случая. Командная часть — на всё приложение (одно на команду), личная — дневник.

---

## Командная часть (+10)

Выполняется один раз на команду. Все условия обязательны. Баллы начисляются **только тем участникам, которые выполнили личную часть.**

1. **Страница 404** — при переходе на несуществующий URL (`/абракадабра`) отображается осмысленная страница, а не белый экран или дефолтная ошибка сервера. Есть ссылка/кнопка назад.
2. **Loading state** — при загрузке данных (API-запрос, переход между страницами) отображается индикатор загрузки: спиннер, skeleton, текст «Загрузка...» — что-то одно.
3. **Обработка ошибок API** — API-запросы обёрнуты в try/catch. При ошибке сети или сервера пользователь видит понятное сообщение, а не белый экран или необработанное исключение.
4. **Видео-доказательство** — короткое видео (~1 мин), демонстрирующее все три пункта выше: 404, loading, error state. Ссылка на видео — в дневнике хотя бы одного участника команды. Голос не обязателен, если из видео всё очевидно.

В разработке вы часто сталкиваетесь с багами — и лучший способ рассказать тиммейтам о баге — записать короткое видео, которое его показывает. Здесь мы просим обратное: не баг, а **доказательство того, что приложение ведёт себя корректно** — отображает 404, показывает loading, обрабатывает ошибки. Так как оценка командная, достаточно, чтобы **один участник** записал это видео за всю команду.

## Как реализовать

### Страница 404

**React Router:**

```tsx
<Routes>
  <Route path="/" element={<Home />} />
  <Route path="/quiz" element={<Quiz />} />
  <Route path="*" element={<NotFound />} />
</Routes>
```

**Angular:**

```typescript
{ path: '**', component: NotFoundComponent }
```

Компонент `NotFound` — простая страница с текстом и ссылкой на главную:

```tsx
function NotFound() {
  return (
    <div>
      <h1>Страница не найдена</h1>
      <p>Проверьте адрес или вернитесь на главную.</p>
      <a href="/">На главную</a>
    </div>
  );
}
```

### Loading state

Минимальный вариант — условный рендер:

```tsx
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);

useEffect(() => {
  fetchData()
    .then(setData)
    .finally(() => setLoading(false));
}, []);

if (loading) return <p>Загрузка...</p>;
```

### Обработка ошибок API

```tsx
const [error, setError] = useState<string | null>(null);

async function loadData() {
  try {
    const res = await fetch("/api/data");
    if (!res.ok) throw new Error(`Ошибка: ${res.status}`);
    const json = await res.json();
    setData(json);
  } catch (e) {
    setError("Не удалось загрузить данные. Проверьте подключение к сети.");
  }
}

if (error) return <p>{error}</p>;
```

## FAQ

_Вопрос:_ Это пересекается с Error Handling в Cross-Check?
_Ответ:_ Частично. W5 даёт 20 бинарных баллов за наличие базовой обработки, Cross-Check (CODE_STANDARDS) даёт до 25 за качество. Они не дублируются: здесь — есть/нет, там — насколько хорошо.

_Вопрос:_ Что если у нас нет API-запросов?
_Ответ:_ Если приложение работает полностью офлайн — условие #3 выполнено автоматически. Покажите 404 и loading для навигации между страницами.

_Вопрос:_ Обязательно ли спиннер? Можно skeleton?
_Ответ:_ Любой индикатор загрузки. Спиннер, skeleton, текст, progress bar — что угодно.

_Вопрос:_ Что если не сделаю чекпоинт?
_Ответ:_ 0 баллов. Других последствий нет, на допуск к защите не влияет.

_Вопрос:_ Это командная оценка — значит один делает за всех?
_Ответ:_ Командная часть одинаковая для всей команды: одно приложение — один набор ошибок. Но дневник каждый пишет сам. Без дневника — 0 за всё.
