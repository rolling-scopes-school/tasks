# RS JS/FE Final Project: Tandem

## 1. Цель проекта

**Цели проекта:**

- Создать приложение для подготовки к техническим интервью и/или тренировки хард-скиллов (JS, TS, Algorithms) в интерактивном формате.
- Получить опыт командной работы.
- Продемонстрировать, чему вы научились на курсе.
- Показать, что вы умеете разбираться в новых технологиях и решать проблемы как Software Engineer.

**Вы можете сделать Tandem в своем стиле** - с той функциональностью, которую вам хочется реализовать. Вот примеры возможных вариантов реализации проекта:

- игра,
- интерактивный тренажер (виджеты, практики, симуляции),
- приложение с LLM / AI Agent / AI-персонажем / и т.д.,
- ваша идея.

Больше деталей о проекте можно прочитать тут:

- [vision_varabei.md](./vision_varabei.md)
- [vision_opus.md](./vision_opus.md)

## 2. Формирование команды

- **Состав команды:** От 2 до 6 человек из числа тех, кто пишет код. Менторы могут писать код и быть полноценными участниками. Рекомендуемый размер команды — 3 человека (при рандомном распределении команды формируются именно так). Команды из 4-6 человек допустимы только при наличии активной менторской поддержки и амбициозного проекта.
- **Team Lead:** Выбирается командой путем голосования. Создает репозиторий проекта на GitHub и выдает доступы остальным участникам.
- **Регистрация:** Каждый участник регистрируется самостоятельно через форму https://forms.gle/UQNknowafoFj1hEt8
- **Каналы коммуникации:** Инструменты для общения (Telegram, Discord и др.) команда выбирает сама.

## 4. Использование AI для написания кода

В этом проекте **разрешено использовать AI** для написания кода. Поэтому абсолютно нормально, если вы будете использовать любые современные инструменты, например, Kiro от AWS (сейчас AWS бесплатно предоставляет 500 Kiro-кредитов на старте) и разработаете часть компонентов с его помощью.  
Главное — прозрачность и честность: отразите это в дневнике разработки и честно укажите, какие части написали вы лично, а какие были сгенерированы LLM-агентом.

## 6. Компонентный подход (Feature Component)

**Feature Component** — самостоятельный функциональный блок, решающий конкретную задачу пользователя (UI-экран, API-сервис, AI-модуль).

**Каждый студент должен лично разработать минимум 2 Feature Component.** Подробнее: [DEVELOPMENT_DIARY.md](./DEVELOPMENT_DIARY.md#требование-2-личных-компонента)

Каждый участник команды выбирает компоненты по своему уровню и интересам.

## 9. Система оценки и процесс сдачи проекта

### Порог допуска к защите финального проекта

- [Дневник](./DEVELOPMENT_DIARY.md) разработки существует, заполнен регулярно, записи закоммичены своевременно
- Описан в дневнике или снят на видео процесс разработки минимум 2 личных Feature Component

### Три обязательных защиты

Все три защиты **обязательны** — без них 0 баллов за соответствующий Score.

1. **[Командная защита](./DEFENCE_TEAM.md)** — команда презентует проект жюри (блоки по 3 команды, ~25 мин). Оценка по критериям [SCORE_TEAM.md](./SCORE_TEAM.md).
2. **[Peer Review](./DEFENCE_PEER.md)** — студент демонстрирует свои фичи 3 студентам-экспертам (~30 мин). Peers валидируют self-assessment. Оценка по критериям [SCORE_PERSONAL.md](./SCORE_PERSONAL.md#3-peer-review-100-баллов).
3. **[Менторская сессия](./DEFENCE_MENTOR.md)** — индивидуальный созвон с рандомным ментором (~30 мин). Чисто технический фокус: код, архитектура. Оценка по критериям [SCORE_PERSONAL.md](./SCORE_PERSONAL.md#4-mentor-review-150-баллов).

- **Weekly Checkpoints (W1–W5)** — по 20 баллов каждый. Помогают держать ритм и не откладывать всё на последнюю неделю. Проверяются автоматически парсером.
- **Tandem: Peer Review** — личная оценка. Студент презентует свою работу 3 студентам-экспертам из других команд (~30 мин). Peers подтверждают или корректируют заявленные баллы из self-assessment.
- **Tandem: Teamwork** — одинаковая для всей команды. Оценивается жюри на презентации: board, git culture, product integrity.
- **Tandem: Personal** — личная оценка каждого студента. Менторская сессия (~30 мин), фокус на коде. Включает Development Diary (100), Personal Features (250 ceiling) и Mentor Review (150). Продукт уже проверен на Peer Review.

## RS Tandem Awards

Помимо основной оценки, лучшие проекты будут номинированы на премию RS Tandem Awards!

### Номинации

1. **Best Application** — Самый целостный, законченный и впечатляющий продукт.
2. **Engineering Excellence** — За самое технически сложное решение, которое команда может объяснить. Не за список технологий, а за глубину: архитектура, нетривиальные проблемы и как их решили.
3. **Best AI Integration** — За самое умное/нестандартное использование LLM (не просто чат, а интересный кейс).
4. **Design Excellence** — За дизайн, анимации и удобство.
5. **Best Presentation** — За лучшую презентацию проекта (демо, storytelling, ответы на вопросы).
6. **People's Choice** — Приз зрительских симпатий.

### Как это работает

**Раунд 1 — Teamwork-блоки.** Команды презентуют в блоках по 3. После каждого блока:

- Жюри номинирует лучших по категориям (не обязательно во всех — сколько заслужили).
- People's Choice: все присутствующие на сессии голосуют за 1 команду (не свою).

**Раунд 2 — Showcase Day.** ~10 команд-номинантов презентуют повторно, все вместе. ~12 мин на команду. Live-презентации (возможно стрим), открытые для всех студентов и менторов. Это шанс для тех, кто пропустил первый раунд, увидеть лучшие проекты. По итогам:

- Жюри выбирает победителей в номинациях 1-5.
- People's Choice — финальное голосование всех участников.

### Tandi's Pick — лучшие дневники

Отдельно от командных номинаций, AI-агент Tandi выбирает **~10 лучших дневников разработки**. Это не конкурс с одним победителем — бейдж получает каждый, чей дневник выделяется качеством: честная рефлексия, описание проблем и решений, эволюция мышления.

### Бенефиты

- Уникальный бейдж в RS APP (Awards и Tandi's Pick).
- Подтверждение достижения для вашего CV/LinkedIn.

## FAQ

### Вопрос: Что может / должен делать ментор в рамках этого проекта?

В отличие от прошлых финальных проектов, в этом проекте ментор может писать код.
Первый вариант — ментор может писать back-end.
Второй вариант — ментор может разрабатывать AI часть. Если студенты тоже хотят разбираться с LLM, то они могут делать свою имплементацию.
Проект допускает наличие нескольких AI Agent с разным поведением (или альтернативной реализацией).

Обязательная активность для менторов — это участие в жюри. Это будет еще одно рандомное распределение, по аналогии с текущими corejs-интервью.
Это можно рассматривать как эволюцию менторинга: сейчас мы активно используем различные инструменты для ревью студенческих работ, поэтому привлекаем менторов и к другим активностям - обучению вместе со студентами, участию в жюри и т.д.

### Вопрос: Почему на протяжении всего курса за использование AI студентов отчисляли, а теперь требуют использовать и/или разрабатывать компоненты, использующие AI?

AI компонент — это один из Feature Component (и даже набор Feature Component), который не является обязательным для всех студентов.  
Прошлые задания были рассчитаны только на личные силы студентов, и использование AI для решения этих заданий не разрешалось.  
В данном проекте мы предлагаем желающим разобраться, как сделать своего AI Agent, как такие системы работают «под капотом» и как их эффективно встраивать в проекты.  
Это можно считать новой базовой технологией — наравне с HTTP или CSS, или умением работать с API.

### Вопрос: Почему командная работа влияет только на часть (±40%) оценки?

Так как в рамках проекта можно делать практически любое приложение, заточенное на подготовку к интервью, то такие приложения уже сложно сравнивать между собой.
Плюс мы разрешаем использовать AI для написания кода. В теории один человек, используя Claude Opus, может заимплементить проект за пару вечеров, но тогда чему-то научится только он один.
Поэтому нас интересует только то, что именно изучил и с чем разобрался конкретный участник команды. В зачёт идут любые знания - всё, что можно назвать инженерией: как классические паттерны, основы баз данных, использование AI и т.д.
Но командная работа всё ещё важна - так как вы придумываете, имплементируете, тестируете общий проект. И его вы будете демонстрировать во время презентаций.

### Вопрос: Как будет осуществляться проверка? Будут ли промежуточные проверки?

Да, есть еженедельные чекпоинты (Weekly Checkpoints W1–W5) — небольшие задания на 20 баллов каждое, которые помогают держать ритм и не откладывать всё на последнюю неделю. Чекпоинты проверяются автоматически парсером. Кроме того, у каждого студента три обязательных защиты: [командная презентация](./DEFENCE_TEAM.md) (блоки по 3 команды), [Peer Review](./DEFENCE_PEER.md) перед студентами-экспертами (~30 мин) и [менторская сессия](./DEFENCE_MENTOR.md) по коду (~30 мин). Все три защиты обязательны — без них 0 баллов за соответствующий Score.

### Вопрос: Можно ли получить сертификат о прохождении курса без выполнения финального задания?

Нельзя. Для получения сертификата студенты должны выполнить финальный проект.

### Вопрос: Могут ли две команды студентов объединиться для выполнения одного проекта?

Да, можно - если менторы команд согласны и у каждой команды будут чёткие сферы ответственности.
Например:
одна команда делает только back-end или AI Agent;
вторая разрабатывает только front-end.

### Вопрос: Что делать, если у меня нет команды?

Сначала попробуйте найти сокомандников самостоятельно — для этого мы создали специальный канал в Discord: #rs-tandem-find-team
Если вы не нашли команду самостоятельно — заполните [форму регистрации](https://docs.google.com/forms/d/e/1FAIpQLScyaWDUyYQBXlPXwrUMmdZDNAGHigBdhGLHzhWLO78TKlFhtA/viewform). Админы распределят вас в команду автоматически.
