# Peer Review

Вариация кросс-чека: каждый студент, дошедший до финалки, **презентует** свою работу трём другим участникам и, в свою очередь, **слушает** три презентации. ~30 мин на сессию.

**Цель:** убедиться, что заявленные в self-assessment фичи реально работают, и потренировать навык презентации.

Все сессии **открыты** — любой студент или ментор может присоединиться как наблюдатель.

## Две роли

Каждый студент в ходе Peer Review выступает в **обеих** ролях:

| Роль             | Что делает                                              | Сколько раз        |
| ---------------- | ------------------------------------------------------- | ------------------ |
| **Презентующий** | Показывает своё приложение, проходит по self-assessment | 1 сессия (~30 мин) |
| **Эксперт**      | Слушает презентацию, задаёт вопросы, оценивает          | 3 сессии (~90 мин) |

**Почему «эксперт»?** Студенты 6 недель делали аналогичное приложение по тому же ТЗ. Они сталкивались с похожими проблемами и могут задавать предметные вопросы, которые не задаст человек со стороны. Кроме того, интересно увидеть, как другие решили ту же задачу, — это и обмен опытом, и тренировка soft skills.

## Формат сессии (~30 мин)

| Этап           | Время   | Что происходит                                                                      |
| -------------- | ------- | ----------------------------------------------------------------------------------- |
| **Контекст**   | 2-3 мин | Студент представляет проект и свою роль.                                            |
| **Demo + Q&A** | 25 мин  | Студент показывает фичи по self-assessment. Эксперты задают вопросы по ходу показа. |
| **Оценка**     | 2 мин   | Эксперты заполняют Google Form.                                                     |

## Принцип валидации

Всё заявленное в self-assessment **считается подтверждённым по умолчанию**. Эксперты отмечают только **расхождения** — фичи, которые не работают, выглядят завышенными или не были продемонстрированы. Если расхождений нет — значит self-assessment подтверждён.

## Оценка (80 баллов)

- Каждый из 3 экспертов ставит оценку независимо (0-80).
- Финальная оценка = **медиана** трёх оценок.
- Если разброс > 30 баллов — координатор проверяет.
- Deploy и артефакты проверяются до защиты на [Week 7 Checkpoint](./WEEK7_CHECKPOINT.md) (+20 pts).

### 1. Feature Demo (+50)

1. (+10) Основной user flow приложения работает — студент проходит ключевой сценарий без критических ошибок
2. (+15) Фичи со ★ из self-assessment работают live и соответствуют заявленному описанию
3. (+10) Дополнительные фичи из self-assessment подтверждаются при показе или по запросу эксперта
4. (+10) Приложение обрабатывает edge cases: 404, loading states, ошибки ввода, пустые состояния
5. (+5) Фичи работают не только на «идеальном» сценарии — эксперт может попросить «а что будет, если...»

### 2. Presentation (+30)

1. (+10) Студент понятно представляет проект и свою роль в команде
2. (+10) Студент уверенно отвечает на вопросы экспертов — объясняет решения, а не только показывает результат
3. (+10) Студент демонстрирует понимание «почему так» — может назвать альтернативы и причины выбора

### Google Form (заполняет каждый эксперт)

Форма **приватная** — студент не видит кто что написал.

1. **Feature Demo** (0-50) — по критериям выше
2. **Presentation** (0-30) — по критериям выше
3. **Что запомнилось из показанного** — свободное текстовое поле
4. **Расхождения с self-assessment** — какие фичи не работали или выглядели завышенными

## Подготовка: презентующий

1. **Self-assessment PR** — создан до сессии ([подробности](./SCORE_PERSONAL.md#self-assessment))
2. **Подготовить демо** каждой заявленной фичи live
3. **Деплой** или быть готовым запустить локально

## Подготовка: эксперт

1. **До сессии (~5 мин на студента):** прочитать self-assessment PR и README
2. **Во время сессии:** смотреть демо, отмечать расхождения с self-assessment
3. **После сессии:** заполнить Google Form
