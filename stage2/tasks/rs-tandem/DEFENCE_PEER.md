# Peer Review

- Каждый студент презентует свою работу трём другим участникам и, в свою очередь, слушает три презентации. ~30 мин на сессию.
- Peer Review - это серия из 4х презентаций
- Период проведения Peer Review сессий с 7 по 21 апреля

**Формальная цель** — подтвердить, что заявленные в self-assessment фичи реально работают, и потренировать навык презентации.

**Настоящая цель** — увидеть, как другие решили ту же задачу. Вы 6 недель писали код в своей команде. Peer Review — редкий момент, когда вы видите три совершенно других подхода к тому же ТЗ. И показываете свой.

Формальную часть вы закроете — пройдётесь по self-assessment, покажете фичи, получите баллы. Но если хотите получить от Peer Review больше — задавайте вопросы: «как ты это сделал?», «почему Supabase, а не Firebase?», «что было самым сложным?» и т.д.

Можно провести презентацию для галочки — список фич, оценка, забыть. А можно рассказать историю: что было, что сломалось, как починили, чем гордитесь.

**Принцип открытости** — любой участник сессии может позвать друзей/команду/ментора, чтобы они присоединились в роли наблюдателя.

## Две роли

Каждый студент в ходе Peer Review выступает в **обеих** ролях:

| Роль             | Что делает                                              | Сколько раз        |
| ---------------- | ------------------------------------------------------- | ------------------ |
| **Презентующий** | Показывает своё приложение, проходит по self-assessment | 1 сессия (~30 мин) |
| **Эксперт**      | Слушает презентацию, задаёт вопросы, оценивает          | 3 сессии (~90 мин) |

**Почему «эксперт»?** Студенты 6 недель делали аналогичное приложение по тому же ТЗ. Они сталкивались с похожими проблемами и могут задавать предметные вопросы, которые не задаст человек со стороны. Кроме того, интересно увидеть, как другие решили ту же задачу, — это и обмен опытом, и тренировка soft skills.

## Экспертные панели

- Студенты распределяются в **панели по 3 эксперта**. Панель формируется рандомно.
- Каждая панель смотрит 3 презентации — можно в один день (~90 мин блок) или в разные дни. Панель решает сама.
- Панель может смотреть только презентации участников из другой панели
- Все сессии должны быть проведены в период с 7 по 21 апреля
- Панель (3 эксперта) связывается со своими 3 презентующими и договаривается о времени
- Сгенерированный список панелей - https://docs.google.com/spreadsheets/d/1uH93ThaKykEu2bRSyFkf5oRYY8Zv0JmFGojSCkf2mBA/edit?gid=0#gid=0
- Если есть проблемы с распределением панелей или другие организационные вопросы — создайте issue: https://github.com/rollingscopes/rs-tandem-observer/issues

## Приблизительный формат сессии (~30 мин)

| Этап           | Время   | Что происходит                                                                      |
| -------------- | ------- | ----------------------------------------------------------------------------------- |
| **Контекст**   | 2-3 мин | Студент представляет проект и свою роль.                                            |
| **Demo + Q&A** | 25 мин  | Студент показывает фичи по self-assessment. Эксперты задают вопросы по ходу показа. |
| **Оценка**     | 2 мин   | Эксперты заполняют Google Form.                                                     |

## Оценка (80 баллов)

Всё заявленное презентующим в self-assessment считается подтверждённым по умолчанию. Эксперты отмечают только расхождения — фичи, которые не работают, выглядят завышенными или не были продемонстрированы. Если расхождений нет — значит self-assessment подтверждён.

- Каждый из 3 экспертов ставит оценку независимо (0-80).
- Финальная оценка = **медиана** трёх оценок. При 2 экспертах — среднее арифметическое.
- Если разброс > 30 баллов — координатор проверяет.

### 1. Feature Demo (+50)

1. (+20) Два личных Feature Component из self-assessment работают live и соответствуют заявленному описанию
2. (+20) Дополнительные фичи из self-assessment подтверждаются при показе или по запросу эксперта
3. (+10) Личные Feature Component корректно обрабатывают edge cases: loading states, ошибки ввода, пустые состояния, неидеальные сценарии

### 2. Presentation (+30)

1. (+10) Студент структурно и понятно объясняет проект, свою роль и личный вклад
2. (+10) Студент отвечает на вопросы по реализации с привязкой к коду/архитектуре, а не только показывает результат
3. (+10) Студент объясняет trade-offs: может назвать альтернативы и причины выбора текущего решения

### Google Form (заполняет каждый эксперт)

**Одна форма на все сессии.** Форма **приватная** — студент не видит кто что написал.

1. **GitHub презентующего** — выбрать из списка
2. **Твой GitHub** — выбрать из списка
3. **Feature Demo** (0-50) — по критериям выше
4. **Presentation** (0-30) — по критериям выше
5. **Что запомнилось из показанного** — свободное текстовое поле
6. **Расхождения с self-assessment** — какие фичи не работали или выглядели завышенными

## No-show и штрафы

Peer Review — это **обмен**: ты оцениваешь других, они оценивают тебя.

**Если эксперт не пришёл на сессию:**

- **-20 баллов** от собственного Peer Review score за каждую пропущенную экспертную сессию
- Пропустил 1 из 3 → максимум 60 pts. Пропустил 2 → максимум 40 pts. Пропустил все 3 → 0 pts.
- Пропуск = не сдана Google Form за данного презентующего.

**Если у презентующего не хватает экспертов:**

- 2 эксперта — сессия **валидна**. Финальная оценка = среднее двух оценок.
- 1 эксперт — связаться с координатором для доназначения.

## Подготовка: презентующий

1. **Self-assessment PR** — создан до сессии ([подробности](./WEEK7_CHECKPOINT.md#self-assessment))
2. **Подготовить демо** каждой заявленной фичи live
3. **Деплой** или быть готовым запустить локально

## Подготовка: эксперт

1. **До сессии (~5 мин на студента):** прочитать self-assessment PR и README
2. **Во время сессии:** смотреть демо, отмечать расхождения с self-assessment
3. **После сессии:** заполнить Google Form (дедлайн: 21 апреля)
