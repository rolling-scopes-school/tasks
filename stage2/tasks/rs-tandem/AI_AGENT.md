## 2. Способы интеграции

### A. Raw Fetch + ReadableStream (Рекомендуется для Technical Depth)

Реализация работы с API "руками" без тяжелых библиотек.

- **Плюсы:** Полный контроль, маленький размер бандла, +баллы в _Technical Depth_ (см. [SCORE_PERSONAL.md](./SCORE_PERSONAL.md)).
- **Что изучить:** `fetch`, `ReadableStream`, `TextDecoder`.

### B. AI SDKs (Vercel AI SDK, LangChain)

Использование готовых библиотек.

- **Плюсы:** Быстрый старт, готовые хуки (например, `useChat` в Vercel AI SDK).
- **Минусы:** "Магия" под капотом, сложнее кастомизировать, если не знаешь базу.

## 3. Mock Mode (Как разрабатывать без боли)

**Проблема:** При разработке UI вы будете обновлять страницу сотни раз. Если каждый раз дергать реальный API:

1. Закончатся лимиты.
2. Будете ждать ответа по 3-5 секунд.

**Решение:** Реализуйте переключатель `IS_MOCK` и возвращайте фейковый стрим.

````typescript
// api/mock-ai.ts
export async function* mockStreamResponse(message: string) {
  const mockAnswer =
    "Это **тестовый ответ** от AI.\n\nВот код:\n```js\nconsole.log('Hello');\n```";
  // Эмулируем, что сеть отдает ответ по кусочкам (чанками)
  const tokens = mockAnswer.split(/(.{1,5})/g);

  for (const token of tokens) {
    await new Promise((r) => setTimeout(r, 30)); // Эмуляция задержки сети
    yield token;
  }
}
````

## 4. Ключевые концепции

### Streaming (Эффект печатной машинки)

LLM генерируют ответ токен за токеном. Не заставляйте пользователя ждать весь ответ 10 секунд. Показывайте текст по мере поступления.

- _Frontend:_ Нужно уметь парсить Markdown, который приходит кусками (иногда разрывая теги).

### Context Management (Память)

LLM не "помнит" предыдущие сообщения сама по себе. Вы должны отправлять ей историю переписки каждый раз.

- _Оптимизация:_ Контекстные окна ограничены (и стоят денег/токенов). Не отправляйте всю "Войну и мир". Обрезайте старые сообщения или саммаризируйте их.

### System Prompts (Personas)

Поведение модели задается системным промптом (первое сообщение в массиве сообщений с ролью `system`).

**Пример "Строгий ментор":**

```json
{
  "role": "system",
  "content": "Ты — строгий сеньор-разработчик. Не давай прямых ответов, задавай наводящие вопросы. Если код плохой — критикуй жестко, но конструктивно."
}
```

### Tool Calling (Advanced)

Современные модели могут не просто генерировать текст, но и "решать", что нужно вызвать функцию.

- _Сценарий:_ Пользователь просит "Проверь мой код". LLM понимает это и возвращает JSON с просьбой выполнить функцию `run_linter(code)`. Ваш бэкенд выполняет функцию и возвращает результат модели.

## 5. Типичные ошибки (Checklist)

| Ошибка                    | Последствия                               | Как исправить                                                                      |
| ------------------------- | ----------------------------------------- | ---------------------------------------------------------------------------------- |
| **API ключ на клиенте**   | У вас украдут ключ и спишут деньги/лимиты | Запросы к LLM только через свой Backend (или Proxy). `.env` на клиенте не спасает! |
| **Нет обработки ошибок**  | Белый экран при падении API               | Оборачивать в `try/catch`, показывать тосты/уведомления.                           |
| **Синхронный Markdown**   | UI "мигает" при стриминге                 | Использовать библиотеки с поддержкой стриминга или `memo` компонентов.             |
| **Нет AbortController**   | Запросы висят, если уйти со страницы      | Отменять фетч при размонтировании компонента.                                      |
| **Отправка всей истории** | Ошибка `Context Length Exceeded`          | Обрезать массив сообщений (оставлять последние N или удалять старые).              |
