# Tandem: Week 2 Checkpoint

> **Score в RS APP:** Tandem: Week 2 Checkpoint
> **Тип оценки:** личная + командная (10 + 10)
> **Дедлайн:** 2 марта 2026, 23:59 UTC
> **Оценка:** дискретная — 0 / 10 / 20 баллов

## Зачем это задание

Week 1 проверял, что команда собралась и начала вести дневник. Week 2 проверяет, что команда **реально пишет код**: проект инициализирован, задеплоен, настроен CI и линтер, задачи распределены через Issues.

Схема начисления баллов намеренно двухступенчатая:

1. Сначала студент выполняет **личную часть** (дневник + назначен на issue) → **+10 баллов**.
2. Только после этого он может получить **командную часть** (деплой + CI + issues + линтер) → ещё **+10 баллов**.

Это отсеивает участников, которые просто числятся в команде, но ничего не делают. Если личная часть не выполнена — командные баллы не начисляются, даже если команда всё настроила.

---

## Личная часть (10 баллов)

Каждый участник выполняет самостоятельно. **Все условия обязательны.**

| #   | Условие                                   | Детали                                                                                                                                                           |
| --- | ----------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **Минимум 2 записи в дневнике за Week 2** | Записи с датами **после 23 февраля**, замержены в `main`. Стандартное правило дневника — 2 записи в неделю (см. [DEVELOPMENT_DIARY.md](./DEVELOPMENT_DIARY.md)). |
| 2   | **Назначен на минимум 1 issue**           | Участник является assignee хотя бы одного issue в репозитории проекта. Это подтверждает, что у студента есть конкретная задача.                                  |

### Как проверяется личная часть

1. **Парсинг `main`:** наличие минимум 2 записей в `/development-notes/{github-username}/` с датами после 23 февраля.
2. **GitHub API:** проверка, что пользователь назначен assignee хотя бы на один issue (`GET /repos/{owner}/{repo}/issues?assignee={username}`).

## Командная часть (10 баллов)

Выполняется один раз на команду. **Все условия обязательны.** Баллы начисляются **только тем участникам, которые выполнили личную часть.**

| #   | Условие                                | Детали                                                                                                                                        |
| --- | -------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **Приложение задеплоено**              | В README есть ссылка на работающий deploy. Ссылка отвечает HTTP 200 и содержит HTML. Даже если это скелет с одной страницей — это уже деплой. |
| 2   | **GitHub Actions workflow существует** | В репозитории есть файл `.github/workflows/*.yml`. CI настроен — даже если пока запускает только линтер.                                      |
| 3   | **Минимум 6 issues созданы**           | В репозитории создано минимум 6 issues. Issues — основной инструмент планирования работы в команде.                                           |
| 4   | **Линтер настроен**                    | В репозитории есть конфигурация ESLint: `.eslintrc.*`, `eslint.config.*` или секция `eslintConfig` в `package.json`.                          |

### Как проверяется командная часть

1. **HTTP ping:** парсер берёт URL деплоя из README → `GET` → проверяет HTTP 200 и наличие HTML.
2. **GitHub API:** наличие файлов workflow (`GET /repos/{owner}/{repo}/contents/.github/workflows`).
3. **GitHub API:** количество issues (`GET /repos/{owner}/{repo}/issues?state=all`).
4. **GitHub API / парсинг:** наличие конфигурации ESLint в репозитории.

## Почему именно эти условия

**Деплой на Week 2** — решает главную проблему финальных проектов: команды откладывают деплой на последнюю неделю, и всё ломается. Если деплой работает на Week 2, дальше только наращивать функционал. Плюс это заставляет решить DevOps-проблемы (CI/CD, environment variables, CORS, build pipeline) пока дедлайн не горит.

**GitHub Actions** — CI это стандарт индустрии. Даже минимальный workflow (запуск линтера на PR) уже приучает команду к автоматической проверке кода.

**6 Issues** — без декомпозиции задач команда не может работать параллельно. 6 issues — низкий порог, но он заставляет хотя бы начать планирование.

**ESLint** — качество кода начинается с линтера. Настроить его на Week 2 значит, что весь последующий код будет проходить проверку.

**Дневник** — продолжение Week 1. Регулярность ведения — ключевой индикатор реального участия.

**Назначение на issue** — простейший индикатор того, что студент знает свою задачу и команда распределила работу.
