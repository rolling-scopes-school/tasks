## Зачем это задание

Week 1 проверял, что команда собралась и начала вести дневник. Week 2 проверяет, что проект инициализирован, задеплоен, настроен CI и линтер, задачи распределены через Issues.

Схема начисления баллов намеренно двухступенчатая:

1. Сначала студент выполняет **личную часть** (дневник + назначен на issue) → **+10 баллов**.
2. Только после этого он может получить **командную часть** (деплой + CI + issues + линтер) → ещё **+10 баллов**.

Если личная часть не выполнена — командные баллы не начисляются, даже если команда всё настроила.

## Личная часть (10 баллов)

Каждый участник выполняет самостоятельно. **Все условия обязательны.**

| #   | Условие                                   | Детали                                                                                                                                                           |
| --- | ----------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **Минимум 2 записи в дневнике за Week 2** | Записи с датами **после 23 февраля**, замержены в `main`. Стандартное правило дневника — 2 записи в неделю (см. [DEVELOPMENT_DIARY.md](./DEVELOPMENT_DIARY.md)). |
| 2   | **Назначен на минимум 1 issue**           | Участник является assignee хотя бы одного issue в репозитории проекта. Это подтверждает, что у студента есть конкретная задача.                                  |

## Командная часть (10 баллов)

Выполняется один раз на команду. **Все условия обязательны.** Баллы начисляются **только тем участникам, которые выполнили личную часть.**

| #   | Условие                                | Детали                                                                                                                                                                                        |
| --- | -------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **Приложение задеплоено**              | В README есть ссылка на работающий deploy. Ссылка отвечает HTTP 200 и содержит HTML. Даже если это скелет с одной страницей — это уже деплой.                                                 |
| 2   | **GitHub Actions workflow существует** | В репозитории есть файл `.github/workflows/*.yml`. CI настроен — даже если пока запускает только линтер.                                                                                      |
| 3   | **Минимум 6 issues в GitHub**          | В репозитории создано минимум 6 issues. Для планирования можно использовать любой инструмент (Jira, Trello и т.д.), но 6 issues должны быть именно в GitHub — проверка идёт через GitHub API. |
| 4   | **Линтер настроен**                    | В репозитории есть конфигурация ESLint: `.eslintrc.*`, `eslint.config.*` или секция `eslintConfig` в `package.json`.                                                                          |

### Как проверяется командная часть

1. **HTTP ping:** парсер берёт URL деплоя из README → `GET` → проверяет HTTP 200 и наличие HTML.
2. **GitHub API:** наличие файлов workflow (`GET /repos/{owner}/{repo}/contents/.github/workflows`).
3. **GitHub API:** количество issues (`GET /repos/{owner}/{repo}/issues?state=all`).
4. **GitHub API / парсинг:** наличие конфигурации ESLint в репозитории.
